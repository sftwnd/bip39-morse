# Как контрибьютить в bip39-morse

*[English version](CONTRIBUTING.md)*

Спасибо, что рассматриваете возможность контрибьюта. Проект маленький
и сфокусированный, поэтому область вкладов чётко очерчена — пожалуйста,
пробегитесь по этому документу перед тем, как открыть issue или PR.

## Способы контрибьютить

- **Баг-репорты.** Открывайте issue по шаблону *Bug report*.
- **Запросы фич.** Используйте шаблон *Feature request*. Пожалуйста,
  держите в голове security-модель инструмента (см. ниже) — фичи,
  которые подразумевают сетевые вызовы, постоянное хранение энтропии
  или внесение случайности, **out of scope by design**.
- **Новые таблицы Морзе по локалям.** Самый естественный путь
  контрибьюта — инструмент построен вокруг подключаемого реестра
  Морзе-таблиц по локалям. Используйте шаблон *New Morse locale*,
  затем присылайте файл таблицы в `examples/<locale>.txt`. См.
  [Добавление Морзе-локали](#добавление-морзе-локали).
- **Документация.** README, README.ru, комментарии в коде, примеры —
  всё приветствуется. Русский README должен быть в синхроне с
  английским для любых пользовательских изменений.
- **Тесты.** Новые тест-векторы, edge-кейсы, fuzz-инпуты для битового
  пайплайна, дополнительные Trezor / BIP39 векторы — всё приветствуется.

## Установка для разработки

Требуется Python 3.11+.

```bash
git clone https://github.com/sftwnd/bip39-morse.git
cd bip39-morse
python -m venv .venv
source .venv/bin/activate
pip install -e ".[test]"
```

Запуск инструмента локально:

```bash
bip39-morse --length 12
python -m bip39_morse --reverse --length 12
```

## Запуск тестов

```bash
pytest
```

Покрытие тестами выставлено на **85%** для модулей `morse`, `bitstream`,
`bip39` и `reverse` (см. `pyproject.toml` → `[tool.coverage.report]`).
TUI-модули (`tui.py`, `tui_reverse.py`) и CLI-точка входа исключены из
coverage, потому что они управляются `prompt_toolkit` и тестируются
через `test_e2e.py` отдельно.

Если ваше изменение трогает битовый пайплайн — пожалуйста, добавьте
тест в `tests/test_bitstream.py` или `tests/test_e2e.py`. Особенно
ценны property-тесты round-trip (forward → reverse → forward должны
воспроизвести тот же entropy hex).

## Добавление Морзе-локали

Репозиторий уже поставляет:

| Файл | Локаль | Стандарт |
|------|--------|----------|
| `examples/japanese.txt` | `jp` | ITU-R M.1677-1 (Wabun) + JARL |
| `examples/greek.txt`    | `el` | ITU-R M.1677-1 |
| `examples/german.txt`   | `en` (слой) | ITU-R M.1677-1 |
| `examples/french.txt`   | `en` (слой) | ITU-R M.1677-1 |
| `examples/spanish.txt`  | `en` (слой) | ITU-R M.1677-1 |
| `examples/polish.txt`   | `en` (слой) | ITU-R M.1677-1 |

Чтобы добавить ещё одну:

1. Создайте `examples/<locale>.txt` в формате, описанном в README
   (заголовок `# locale: <code>`, затем строки `<char> <whitespace> <morse>`).
   В верхнем комментарии процитируйте использованный стандарт.
2. Если файл задаёт *новый* алфавит (например, греческий, арабский,
   иврит) — присвойте ему собственный код `locale:`. Если он только
   добавляет акцентированные буквы к существующему алфавиту (например,
   польский поверх латиницы) — используйте код базовой локали, чтобы
   записи *наслаивались* на встроенную таблицу.
3. Добавьте кейс в `tests/test_morse_tables.py`, который покрывает хотя
   бы один round-trip через новый файл.
4. Добавьте короткий параграф в раздел *Custom Morse tables* в README
   (и в русский README) — минимум: цитируемый стандарт, добавленные
   символы, любые caveats.
5. **Важно:** инструмент сейчас полагается на то, что для каждого бита
   есть однобитная буква, которая его декодирует (`e`/`т` покрывают
   `0`/`1`). Если ваша локаль не имеет однобитных кодов, fallback-цепочка
   `bits_to_text` (буквы → цифры → пунктуация) всё равно должна покрывать
   `0` и `1` на какой-то длине; проверьте это round-trip тестом перед
   отправкой.

## Стиль кода

- Синтаксис Python 3.11+ приветствуется (match-выражения, type unions
  через `|`).
- Type hints на публичных функциях желательны.
- Без новых runtime-зависимостей, если только это не абсолютно
  необходимо. Сейчас runtime-footprint — только `prompt_toolkit`, и мы
  хотели бы это сохранить.
- Держите модули маленькими и single-purpose; повторяйте существующий
  layout (`morse.py` для таблиц, `bitstream.py` для битовой
  арифметики, `bip39.py` для wordlist/мнемоник, `reverse.py` для ввода
  слов, `tui*.py` для UI).

## Сообщения коммитов

Императив, настоящее время, первая буква строчная, без точки в конце
заголовка; тело сообщения переносить около 72 символов. Одно логическое
изменение на коммит; разделяйте рефакторинги и изменения поведения на
отдельные коммиты, где это возможно.

Примеры из текущей истории:

```
Add --group-size and --per-line for reverse output formatting + CI
Make demo flow inline-realistic: TUI redraws in place, output appends below
Add iroha terminal demo (animated SVG + static PNG) to READMEs
```

## Процесс pull request

1. Сделайте форк репозитория и создайте topic-ветку от `master`:
   `feature/<short-name>` или `fix/<short-name>`.
2. Запустите `pytest` локально — должен пройти с coverage ≥ 85%.
3. Обновите README (и `README.ru.md`) для любых пользовательских
   изменений.
4. Добавьте запись в `CHANGELOG.md` под `[Unreleased]`.
5. Открывайте PR против `master`. Заполните PR-шаблон.
6. CI (`.github/workflows/tests.yml`) запустит pytest на push и PR.
   PR должен быть зелёным до начала ревью.

Обычно отвечаю в течение нескольких дней. Если PR висит без обратной
связи неделю — спокойно пингуйте.

## Security-чувствительные контрибьюты

Это инструмент, который работает с материалом, напрямую завязанным на
seed-фразы кошельков. Некоторые категории контрибьютов требуют
повышенной аккуратности:

- **Не вставляйте** настоящие мнемоники, настоящую энтропию или
  настоящие адреса кошельков в issues, PR, скриншоты или тестовые
  фикстуры. Используйте уже лежащие в репозитории ироха-векторы,
  публичные тест-векторы Trezor или свежесгенерированные одноразовые
  фразы.
- **Не предлагайте** изменения, которые вносят сетевые вызовы,
  телеметрию, запись в буфер обмена, постоянное хранение энтропии
  на диске или внесение случайности. Это намеренные non-features
  (см. раздел *Security notes* в README).
- Для всего, что затрагивает `bitstream.py`, `bip39.py`, `morse.py`
  или `reverse.py`, пожалуйста, включайте round-trip тест.
- Если вы нашли security-уязвимость — **не открывайте публичный issue**.
  См. [SECURITY.md](SECURITY.md) — там приватный канал для
  репорта.
